# Upstream Merge Policy

> 适用仓库：`iolh/remotelab`  
> 目的：只合并真正提升可落地性、或真正解决当前痛点的 upstream 改动。

## 核心原则

不追“最新”，只追“有用”。

默认判断标准只有两条：

1. 是否提升这条产品线的可落地性
2. 是否解决当前真实痛点

如果答案都是否定的，即使 upstream 看起来更先进，也不默认合并。

## 维护哲学

这条分支已经形成了自己的产品方向：

- 主线 / 验收 / 裁决的能力分工
- typed handoff
- 主线吸收区
- 只读验证运行时

因此长期策略不是“紧跟 upstream”，而是：

- **基础设施跟 upstream**
- **工作流保持本地判断**

## 默认应评估并优先合并的改动

这类改动通常会降低维护成本，或提升系统稳健性：

- 安全修复
- 认证与权限修复
- 配置目录 / memory 目录 / 实例隔离
- 部署脚本、启动逻辑、恢复逻辑
- 通用 runtime 兼容与适配
- 数据持久化与兼容性修复
- 明显的 bugfix

这类改动的判断原则：

- 如果不会明显冲击本地工作流哲学，优先合并
- 如果有冲突，优先保留 upstream 基础设施能力，再局部适配本地产品线

## 条件性合并的改动

这类改动只有在真实使用场景中高频出现时才值得合：

- 语音输入增强
- composer / editor 体验优化
- 分享 / 快照 / 访客类增强
- 低频 UI 优化
- 移动端体验增强

判断原则：

- 高频使用 → 值得评估
- 低频使用 → 默认暂缓

## 默认谨慎或暂不合并的改动

这类改动最容易冲淡本地工作流设计：

- system prompt 哲学
- continue bias / 自主推进偏置
- agent 默认行为增强
- 会话语义或主线边界变化
- 会改变主线/辅助线关系的产品交互

判断原则：

- 不因为 upstream 官方采用就默认合并
- 只有当它明确改善本地工作流，且不会破坏既有主张时才考虑

## 评估流程

每次看 upstream 改动时，按下面顺序判断：

1. 这条改动属于哪一类？
2. 它解决的是基础设施问题，还是工作流哲学问题？
3. 它是否直接解决当前痛点？
4. 它是否会与以下本地能力冲突？
   - `主交付 / 执行验收 / 深度裁决`
   - typed handoff
   - 主线吸收区
   - 只读验证运行时
5. 冲突成本是否值得？

## 当前已确认的案例

### 建议合并

- `96df5ea` `add isolated instance root support`

原因：

- 属于基础设施增强
- 提升实例隔离和长期维护能力
- 与本地工作流主张冲突小

### 条件合并

- `b4be9a4` `add session-aware voice transcript cleanup`

原因：

- 如果高频使用手机语音，价值明显
- 如果不是高频语音用户，优先级中等
- 与本地前端改动冲突较多，合并成本不低

### 暂不合并

- `edfe46a` `increase default continue bias in system prompt`
- `290eff6` `tighten self-check continuation bias`

原因：

- 不直接解决当前痛点
- 容易扰动本地已经建立的主线/验收/裁决工作流
- 属于“行为哲学层”改动，默认谨慎

### 可忽略

- `f961a9f` `promote auth/visitor/composer learnings`

原因：

- 主要是 memory / learnings 调整
- 对当前产品线的直接价值低

## 一句话版

长期策略不是“跟不跟 upstream”，而是：

> 只吸收能增强基础设施和真实可落地性的改动；  
> 不为了“看起来更新”而牺牲自己的工作流设计。
