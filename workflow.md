## 版本线

### v1 — 按阶段划分辅助入口

三入口：`主交付` / `风险复核` / `PR把关`

- `主交付`：Codex + GPT-5.4，唯一主线
- `风险复核`：Opus 4.6，方案形成后或改动完成后做高压缩风险扫描
- `PR把关`：Opus 4.6，PR 评论归并、冲突裁决、合并前 gate

**问题**：`风险复核` 混合了执行验证和深度判断两类不同能力；`PR把关` 按阶段命名，与按内容命名的 `风险复核` 分类维度不一致。

### v2 — 按能力角色重新划分

三入口：`主交付` / `执行验收` / `深度裁决`

核心变化：
- 验证类工作归 GPT-5.4（执行验收），不再浪费 Opus 算力
- 判断类工作归 Opus 4.6（深度裁决），聚焦高不确定场景
- 引入 typed handoff 协议概念
- 定义交付物合同：每个入口有明确的输入/输出/不负责/收口标准

### v2.1 — 补齐实现边界（当前有效规格）

在 v2 基础上补齐 5 条实现边界：
- 同模型入口的运行时硬隔离
- handoff 覆盖关系与 `superseded`
- 深度裁决的冷启动输入协议
- 执行验收 session 的复用策略
- 最简路径

实现状态（截至 2026-03-22）：
- [x] typed handoff（`verification_result` / `decision_result`）
- [x] `superseded` 覆盖规则
- [x] 空状态 prompt 注入
- [x] 执行验收只读运行时
- [x] payload 结构化（validated / unverified / findings / evidence / recommendation / confidence）
- [ ] 半自动编排（Phase 2）
- [ ] 自动挡入口（v3 愿景）

### Phase 2 — 半自动编排（下一步）

在 v2.1 已落地能力上，增加：
- 主线 ready 信号
- 系统提示 + 用户确认
- 自动创建辅助线 session
- 结果自动回流

详见 [`workflow-phase2-plan.md`](workflow-phase2-plan.md)。

### v3 — 自动挡入口（愿景）

- 单一任务入口 + 信息采集卡
- 模式推荐引擎（快速执行 / 标准交付 / 审慎模式 / 并行推进）
- 用户确认后系统全自动编排
- 手动模式降为高级入口

v3 依赖 Phase 2 积累的使用数据和半自动编排的可靠性验证。

## 设计决策备忘

### 为什么按"能力角色"而不是"阶段"划分

`PR把关` 不是一种能力，而是一个阶段。到了 PR 阶段，真正需要的是两种东西之一：验证真实行为（执行验收）或做复杂判断（深度裁决）。按能力角色划分后，用户的判断路径从"我在哪个阶段"变成"我需要什么能力"，维度一致。

### 为什么执行验收用 GPT-5.4 而不是 Opus

验证类工作的核心是"真实执行 + 证据采集"：跑测试、看页面、查边界。这些依赖 agent 的工具调用能力和环境联动能力，而不是深度推理能力。GPT-5.4 在执行维度更强。

### 为什么先做 result 再做 request

用户现在已经能手动开辅助线（request 的人工替代成本低），但手动把辅助线结论搬回主线很痛苦（result 的人工替代成本高）。先自动化替代成本高的那端。

### 为什么跨源同类结果不自动覆盖

当前系统里没有稳定的 `taskId` 概念。为了覆盖关系引入新的任务主键会膨胀 MVP 范围。保守策略：默认鼓励复用同一个辅助 session，跨源场景由用户决定。
