# RemoteLab Workflow Review Notes

## 目的

这份文档用于给 Claude Code 审阅当前这套基于 RemoteLab 的 AI 协作工作流。

目标不是宣传，而是实事求是地说明：

- 当前工作流到底是什么
- 它为什么这样设计
- 它目前已经做到什么程度
- 它还缺什么
- 接下来最值得改的是什么

## 一句话结论

当前这套工作流已经具备一个优秀 AI 协作工作流的明确雏形，而且已经可操作。

它的核心优点不是“多模型都能接”，而是：

- 用户输入已经从“命令式 prompt”收敛成“事实 / 观点 / 决策”
- 角色分工已经清晰
- 主线与辅助线已经区分开
- 并行开发已经开始引入 `git worktree` 作为代码隔离手段

但它目前仍然更像一个 **强雏形 / v1.5**，而不是成熟的“系统级工作流产品”。

最主要的原因是：

- 关键协作关系仍然有一部分靠提示词和人工纪律维持
- 还没有完全产品化成系统默认行为

## 当前工作流结构

### 三个核心入口

1. `主交付`
   - 唯一主线
   - 默认承担执行计划、实现推进、收口、最终可验收输出
   - 运行时：`Codex + gpt-5.4 + medium`

2. `风险复核`
   - 辅助线
   - 用于在方案形成后、改动完成后或合并前做高压缩风险扫描
   - 运行时：`Cursor + claude-4.6-opus-high-thinking`

3. `PR把关`
   - 辅助线
   - 用于 PR 评论归并、冲突裁决、合并前 gate
   - 运行时：`Cursor + claude-4.6-opus-high-thinking`

### 工作流原则

- 永远只有一个主线：`主交付`
- `风险复核` 和 `PR把关` 不能演化成第二个主开发线程
- 用户默认只需要长期盯 `主交付`
- 用户主要提供：
  - 事实
  - 观点
  - 决策
- 输出格式、推理步骤、收口规则尽量由 App 提示词承担

### 典型顺序

1. 新需求进入：先开 `主交付`
2. 有方案或已有一轮代码：开 `风险复核`
3. 评论回来或准备合并：开 `PR把关`
4. 结果统一回到 `主交付`
5. 最终由 `主交付` 输出：
   - 改了什么
   - 为什么这么改
   - 风险点
   - 怎么验收
   - 验证证据

## 为什么是这三个入口

### 这套命名比“模型人格”更接近真实工作

这三个名字不是按模型能力命名，而是按工作时机和职责命名：

- `主交付`：新需求和最终收口都在这里
- `风险复核`：有方案或改动后再来
- `PR把关`：明确只在 PR 阶段使用

这让用户更容易判断“什么时候开哪个入口”，而不是记住某个模型擅长什么。

### 模型分工是互补的，不是对抗的

- `Codex` 更适合主线实现、真实改代码、推进任务
- `Opus` 更适合挑风险、补边界、归并评论、做 go/no-go 判断

因此当前设计不是“永远先 Opus 再 Codex”，而是：

- 主线默认 `Codex`
- 判断型线程默认 `Opus`

这更符合任务实际，而不是为了“看起来更高级”强行串模型。

## 用户侧交互是否优雅

### 已经做到的部分

从提示词设计角度看，用户侧已经比较优雅：

- 用户不再需要反复教 agent “你应该怎么输出”
- 用户不再需要每次都发“请先分析、再分点、最后总结”
- 用户输入已经接近真实工作表达，而不是 prompt engineering

这一点我认为是当前工作流最重要的进展之一。

### 还没完全优雅的部分

虽然输入方式已经优雅，但工作流机制还没完全内化：

- 用户仍然需要手动决定开哪个入口
- 用户仍然需要一定程度上理解主线 / 辅助线关系
- 并行和收口机制还没有完全变成系统默认动作

所以它目前是：

- 用户侧：已经比较优雅
- 系统侧：还在从“提示词驱动”走向“机制驱动”

## 并行开发能力现状

### 已经具备的能力

当前并行不是“一个对话里同时做很多事”，而是：

- 一个主交付会话
- 必要时派生实现子线程
- 辅助线程做风险复核
- 后续再做 PR 把关

为了让并行写代码更可控，已经额外引入了 `git worktree` 工作流：

- 一个实现子线程对应一个独立 `worktree + branch`
- 通过本地脚本创建和回收 worktree

这解决的是“工作区冲突”问题，而不是抽象的多线程问题。

### 当前边界

并行能力是可用的，但还不是完全系统化的：

- RemoteLab 负责 session 级并行
- `git worktree` 负责代码工作区隔离
- 这两者目前是“工作流组合”，不是“产品内建一体化编排”

换句话说：

并行开发已经能务实落地，但还没有变成系统自动维护的主能力。

## 刚完成的最小产品化：结果回灌

本次已经把工作流从纯提示词约束，往产品机制推进了一步。

### 新增能力

在 `风险复核` / `PR把关` 中，可以把当前会话的最新结论一键“回灌”到 `主交付`。

当前行为：

- 辅助会话顶部出现 `回灌` 按钮
- 第一次回灌时选择目标主线
- 之后记住目标主线
- 主线收到一张结构化回灌消息卡

### 这一步的价值

它解决的是当前最高频、最机械的人工动作：

- 用户不用再手工复制粘贴辅助线程结论
- 主线第一次真正开始承担“结论汇流点”的角色

这不是最终形态，但它是从“靠提示词和纪律”走向“系统开始帮你维持工作流”的第一步。

## 当前成熟度判断

### 我的判断

这套工作流当前更适合被定义为：

- **强雏形**
- **已经可操作**
- **值得在真实任务里持续使用**
- **但还没到成熟产品工作流**

如果粗略打分：

- 方向正确性：8.5 / 10
- 当前可操作性：7.5 / 10
- 系统成熟度：6.5 / 10

### 为什么不是更高

因为现在还有几类关键能力没有完全产品化：

1. 主线状态面板还没有落地
2. 子线程登记和边界展示还没有落地
3. 外部输入源（如飞书 / Bitbucket）还没有真正收编
4. 并行调度仍然需要较多人工纪律

## 当前最主要的优点

1. 用户输入负担显著下降
2. 主线 / 复核 / PR 的职责边界足够清晰
3. 模型分工合理，互补而不重叠
4. 工作流命名基于任务时机，而不是模型人格
5. 已经开始把关键协作动作产品化，而不是只停留在 prompt 层

## 当前最主要的风险

1. 仍然依赖用户理解并遵守工作流边界
2. 结果回灌已做，但主线状态区还没做，接管效率仍有提升空间
3. 并行开发虽然有 `worktree`，但没有 session 内建登记和可视化
4. 这套工作流现在最适合 1 个主线 + 少量辅助线，尚不适合高密度并行

## 我认为最值得继续做的三件事

### 1. 主交付状态区

让 `主交付` 更像控制面板，而不是普通聊天历史。

最少应该固定展示：

- 当前目标
- 当前阶段
- 已完成
- 当前风险
- 下一步
- 待决策项

### 2. 子线程登记

让并行线程从“散落 session”变成主线可管理的工作单元。

最少应该登记：

- task
- worktree path
- branch
- allowed paths
- child state

### 3. 更自然的回灌吸收

现在已经能一键回灌，但主线还没有“待处理结论”区。

下一步可以考虑：

- 把回灌卡片标记成待处理结论
- 在主线里显式区分“普通消息”和“待处理工作结论”

## 希望 Claude Code 重点审阅什么

请优先从下面几个角度看：

1. 当前三入口的边界是否足够清晰，是否还存在明显职责重叠
2. `主交付 = Codex`，`风险复核 / PR把关 = Opus` 这套模型分工是否合理
3. 当前“用户主要只提供事实 / 观点 / 决策”的输入哲学是否已经成立
4. `结果回灌` 这个 MVP 是否是正确的最小产品化切入点
5. 接下来应优先做“主线状态区”还是“子线程登记”
6. 如果要把这套工作流从强雏形推进到成熟系统，最小下一步应该是什么

## 最后的务实判断

这套工作流已经不是“想法”，而是一个能真实跑起来的系统骨架。

它最难的部分不是再写更漂亮的 prompt，而是继续把这些已经证明有效的协作关系做成产品机制。

所以我对它当前阶段的定义是：

> 一个已经可用、方向正确、值得持续投入的 AI 协作工作流雏形；
> 它的下一个关键门槛不是提示词优化，而是工作流产品化。
